inhouse 埋点设计

2021-09-04 回顾, 埋点, 方案设计

inhouse 埋点方案设计

项目背景

目前公司已经拥有多套第三方的 tracking 系统，比如 GA，Mixpanel 等，主要是为了满足 marketing 和 product analytics 等方面的分析需求。随着业务的飞速发展，GA 和 Mixpanel 不断地增加新的埋点，两套系统已经变的非常复杂，而且已经存在不少数据质量的问题。

搜索和推荐目前正在朝着大数据驱动的智能平台方向发展，对数据的采集有了新的需求。搜索和推荐将以业务的核心指标作为优化的最终目标，比如 CTR，CVR，GMV 等；实现这些优化的重要手段将是机器学习。而机器学习需要大量的数据，这些数据通过现有的第三方 tracking 系统是很难获取的。比如：

  • 曝光数据是计算核心指标 CTR 的重要数据源，但目前各个第三方平台并没有采集。
  • 埋点原始数据可以增强数据基础能力，让更深度的分析成为可能，但第三方平台不一定能提供原始数据。

因此我们打算开发一套内部的 tracking 系统，采集我们所需要的一切用户数据，上报到我们自建的大数据平台中，为实现数据驱动和平台智能化提供基础数据能力的支持。

此外，旧的埋点方案存在一些问题，导致埋点效率低、数据质量不稳定。

对于产品或其他数据使用方来说：

  • 埋点口径不一致、不规范
  • 埋点数据不准确
  • 埋点数据难以验证

对于前端开发来说：

  • 来源路径都需要手动埋点，成本较高
  • 前端需要自行通过接口拉取额外的属性数据
  • 需要处理服务端字段名与第三方埋点字段名的映射关系
  • 模块复用容易导致埋点代码逻辑出错
  • 埋点没有规范，特殊逻辑较多
  • 通过 in-house tracking 项目，我们希望在框架层面上解决这些问题，使新的埋点系统可以做到高效率、高质量、可验证与可持续。

整体设计目标

  1. 前端埋点：支持核心指标分析
  • 准确并且精细的追踪每一条用户流量路径，能够准确的复盘每一个订单的来源
  • 支持 CTR、CVR、GMV 等核心指标的计算
  1. 前端埋点：标准化和框架化
  • 保证埋点数据的准确性
  • 减少研发埋点成本
  • 兼容第三方 tracking 系统
  • 支持实时将 in-house 的埋点数据按需转化并转发到第三方，比如 Mixpanel 和 GA
  • 与 Kepler AB 实验系统结合
  • 有效的标识每个实验所影响的流量和数据，包括前端实验和后端实验
  • 准确的衡量分析实验的指标及效果
  1. 支持后端埋点
  • 后端可按需发送埋点 log，可与前端埋点协同分析
  • 规范数据测试流程
  • 埋点数据测试流程标准化和自动化，支持大面积埋点需求测试
  • 埋点数据测试纳入回归测试中，保证新需求不影响埋点数据

核心数据关联

  • session_id
    聚合同一个用户在一个连续的时间周期内的行为
    是用户行为分析和核心指标计算的一个重要的概念
  • page_open_id
    聚合同一个用户在同一次页面打开中的行为
    前后端数据的重要连接器
  • click_id
    由点击事件生成并传递给点击所打开的页面
    前后页面之间的连接器，使深度的路径分析成为可能

前端埋点

  1. 元素编码化
    借鉴阿里的 SPM (super position model) 模型，平台统一使用一套编码规范来定位页面上的每一个元素。SPM 编码由 页面编码、模块编码和元素编码 三级构成。
  2. 编码与页面元素绑定
    通过声明式标记配合 UI 页面元素的层级结构来埋点的方式, 减少埋点对于业务代码的入侵。
  3. 事件类型
  • pageview 事件：页面级别，用于追踪页面流量
  • exposure 事件：模块级别，使用有效曝光数据计算 CTR
  • action 事件：模块或者元素级别，默认是 click 事件，后续会有 input、hover 等事件
  • custom 事件：自定义事件类型。无法被以上三种标准化事件包括的事件，比如要求在页面拿到某个接口数据之后再触发
  1. 数据关联
  • unique_event_id：每一个事件都会生成的 UUID，用于标记每个事件，便于后续对数据进行去重。
  • page_open_id：每一次页面打开时生成的 unique ID，请求时发送到后端服务。用于聚合同一个页面上发生的所有事件，以及关联前端埋点与后端埋点。
  • 对于 pageview 事件来说，page_open_id 等于 unique_event_id。
  • 由于 page_open_id 是 page 级别的属性，会被模块和元素级别的事件继承，因此 exposure 和 action 事件上都会包含其对应的 pageview 事件的 page_open_id。
  1. click_id：每一次用户点击时生成的 unique ID，页面跳转时发送到下一页面。用于关联前后两个页面跳转，提供深度路径分析能力。
  • 对于 action（click）事件来说，其 click_id 等于 unique_event_id。
  • 由于 click_id 会被传递到点击跳转的下一个页面，并且被放在下个页面的 page 级别属性上，所以跳转的页面上的所有事件的 page.click_id 都会记录此点击的 click_id。
  1. session_id：frontLogSrv 根据维护的 session 生成的 unique ID。用于聚合同一用户的一段连续行为。

数据转发服务

数据转发服务是 in-house 前端埋点数据与 Mixpanel/GA 等第三方埋点工具的数据中转站，主要负责数据补全和数据转换/转发两大功能模块。

数据补全

前端直接发送的埋点数据遵循尽量精简的原则，一般只发送必要的、只能由前端获得的数据，其他可由后端补全的数据则会在这一阶段进行补全，补全后的数据会发到大数据平台作为 in-house 的数据进行后续处理，同时也会进入下一阶段进行数据转换。
数据转换和转发
数据转换和转发模块主要服务于第三方埋点工具，会根据 Mixpanel/GA 等第三方的需要对数据进行转换，然后发送到对应的第三方平台。
大数据平台数据补全
在 IHT 系统运行一段时间后，发现由后端服务来进行数据补全有一定的局限。因此我们运用大数据平台的能力，将数据补全功能从转发服务中抽离开来。具体设计详见数据补全方案文档。

Kepler AB 实验信息

Kepler 同时支持前端和后端的实验。在 in-house 的整体方案中，我们希望可以结合 AB 实验的数据来对用户行为进行分析，同时也希望 in-house 的数据可以作为 Kepler AB 实验结果分析的一个出口。
前端实验
前端发送埋点数据时，会将用户命中的所有实验的分组结果加入到每一个事件中。因此对于前端实验，我们通过前端埋点数据即可对比不同分组的用户数据。
后端实验
后端进行实验处理时，通过 Kepler SDK 查询用户命中实验信息时，会使用 Kepler 封装的 Logging 模块，上报用户命中的实验信息，以及可选的业务自定义字段（比如模型打分结果）。在进行实验分析时，通过 Kepler Logging 上报的后端数据，结合通过 page_open_id 连接到的前端埋点，即可计算出实验指标。
其他后端 log
除了需要进行 AB 实验的模块之外，后端服务的其他模块可能也会需要进行埋点。比如搜索服务可能需要记录用户的搜索词，以便进行用户搜索词的词频分析。这些数据会直接发送到统一日志上报服务中。
统一日志上报系统
前述的三种数据（in-house 的前端埋点数据、Kepler AB 实验的埋点数据，以及后端埋点数据），都会通过基础架构的统一日志上报系统进行上报，并通过 kafka 集群最终存储到 S3 桶中。

前端埋点
SPM 定义和规范
SPM 编码会包括页面、模块、元素三个层级，以{pageCode}.{moduleCode}.{itemCode}的形式拼接为最终的 spm 编码。元素归属于某个模块，而模块会归属于某个页面；但页面与页面之间、模块与模块之间、元素与元素之间都不会存在层级关系。
页面、模块和元素与代码中的概念或者产品定义的概念都没有耦合关系, 它们的定义和映射关系主要由文档来维护。

定义

规范

  • Page
    页面
    页面是一个或者多个模块的集合。
    一般 web 上的一个 URL path，以及 APP 上的一个全屏的界面，就是一个页面。有时候为了方便用户操作，我们会在某些页面上弹出浮层来展示比较大的一块内容，用户交互完后就会关闭浮层返回之前的页面。这些浮层可能会也可能不会被定义为页面，取决于埋点需求方的定义。
    页面层级上只发送 pageview 事件，主要用来进行流量分析和用户路径分析。
    页面一般是对站点的整体流量做出贡献，或者对用户跳转路径有影响的模块集合
    页面内容一般会超过一个屏幕的大小
    页面作为一个整体被加载，当页面被加载时，其中的每个元素不一定都已经被用户看到
    Module
  • 模块
    模块是页面的一部分，是一个或多个元素的集合。
    模块是在功能和位置上比较相近的元素的虚拟集合，一般可以对应到 UI 上的一个线框。模块是计算曝光的最小单元，一般不应超过屏幕大小。
    模块层级上会发送 exposure 事件和 action 事件，主要用于计算模块本身或模块中元素的 CTR。
    模块一般是作为一个整体被曝光的元素集合，一般不应该超过屏幕大小
    当一个模块被曝光时，我们认为其中的每个元素都已经被用户看到。当一个浮层在逻辑上既可以是页面也可以是模块的时候，这里可以作为主要的区分点
    部分列表式的模块（比如首页热门推荐中的多个活动卡片）在命名上会统一使用_LIST 作为后缀，此时这个模块名是指这个列表中的每一个项，埋点数据中会再发送 list_len 和 list_index 字段来区分不同位置的项
    模块支持发送 action 事件，但一般只针对没有显式地定义所含元素的模块发送。对于有显式定义元素的模块，则应该对元素而不是对模块发送 action 事件
  • Item
    元素
    元素是模块的一部分。
    元素是最小可交互单位，一般来说，是一个按钮或者链接。
    元素层级上会发送 action 事件。结合其所在的模块的曝光，可用于计算元素 CTR；通过 click_id 向下一个页面的传递，可用于连接用户在不同页面之间的跳转路径。
    元素一般包含了具体的交互功能
    元素自身没有是否曝光的属性
    当将界面 UI 定义为元素时，即是将其作为埋点上关注的最小单位，元素中不再包含其他可追踪的单位

配置化页面的 spm

对于配置化的页面，为了在前端内容动态下发的同时，能够保证 spm 对于前端页面元素的唯一性和描述性，我们结合配置化页面接口的结构对模块名做了特殊设计。这些模块名会遵循特定格式：sectionName:sectionType:tabName:dataType:dataName

事件类型

页面依赖的接口失败的情况暂时不上报
页面打开来源：对于从 deeplink 进入和系统通知唤醒的情况，需要通过页面属性中的 app_open 字段标记。需要注意 Android 和 iOS 的统一；Android 上对于 deeplink 的中转页面不应该记录 pageview
搜索起始页作为一个特殊的页面，可能在 APP 的多个页面上被打开。在不同页面上被打开时，具体处理逻辑可能不同，而从数据分析的视角也希望将不同来源的页面区分开，因此这个页面做了特殊处理，会将前一个页面的编码加入到搜索起始页中作为区分{previousPageCode}_SearchStart。这个特殊逻辑由于实现逻辑复杂容易出错，暂时不做。
exposure 事件
exposure 事件只在模块上触发
有两个阈值设置，需要达到曝光事件的阈值才需要发送曝光
曝光时长阈值：默认 0.5
曝光面积阈值：默认 80%
曝光事件开关：不是所有定义的模块都需要追踪曝光，只需要对需求文档上定义的模块进行埋点即可
曝光捕获性能：客户端对于曝光捕获的性能需要进行进行关注，需要保证捕获性能不会影响到用户的体验
模块在页面上交叉超过曝光面积阈值时算曝光开始，模块开始曝光之后，模块和页面交叉低于曝光面积阈值算曝光结束，duration = 曝光结束时间 - 曝光开始时间
特殊处理：有一些 exposure 事件需要在列表刷新的时候进行记录，比如搜索 suggest 列表，用户持续输入的过程中列表的内容会进行刷新，刷新时需要重新计算 exposure，在 SDK 设计上需要考虑到这样的情况
action 事件
action 事件可能在模块上也可能在元素上触发
action 表示用户直接的操作，可能是 click、input、slide 等，目前大部分情况下都是 click 事件
对于 click 事件，每次用户点击时都需要生成一个 unique 的 click_id，并将 click_id 和 click_spm（当前元素的 spm）传递到下一个打开的页面
部分页面里面可能有可操作元素，但是没有定义模块，这时候在模块编码的位置上记录为 DefaultModule，便于后续 spm 的切分统计
custom 事件
custom 事件可以跟页面绑定，也可以不跟任何页面绑定。跟页面绑定时，spm 的 page 编码等于绑定的页面的编码；而 module 编码可以自定义。
custom 事件会继承 page 级别的属性。
click_spm 和 click_id
click_spm 和 click_id 在进行分析的时候非常重要，可以告诉我们不同模块的转化表现。但是当埋点不全、埋点数据丢失、用户行为链路复杂等多种因素作用下，正确记录 click_spm 并不那么容易。
APP 设计方案
当前 APP 端对 click_spmn 和 click_id 的处理逻辑是（经讨论后修改的版本，详见 orion 工单）：
对于每个 pageview 事件，APP SDK 会在已发生的事件队列中寻找最后一个 action/custom/pageview 事件。
找到对应的事件后，获取该事件的 unique_event_id 作为 page.click_id，获取该事件的 event_common.spm 作为 page.click_spm。
如果找不到对应的事件，则 page.click_id=-1, page.click_spm=unknown。
这个设计是为了在寻找 pageview 的来源 click_id/click_spm 时，可以只基于事件流，而不需要处理其他特殊逻辑，减少出错的可能。
之所以关联 custom 事件，是因为一些 pageview 可能是由于 AppLaunch、AppOpen 等 custom 事件（见上面的新增 event 需求）而打开的，我们希望此时的 click_spm 记录的是 AppLaunch、AppOpen，而不是上一个点击事件。
之所以关联 pageview 事件，是因为当页面埋了 pageview 的点而没有埋点击事件时，跳转到下一个页面仍然可以记录到是从哪个页面点击来的。随着我们逐渐提高 pageview 埋点的覆盖率，这个也会在很大程度上提高 click_spm 的准确性（减少关联到上上个页面的情况）。
这个方案可能存在的问题是：
当埋点不全时，pageview 可能会关联到不正确的事件。如 A-B-C，若 B 没有埋点，则 C 页面的 click_id/click_spm 会找到 A 页面上的点击事件。
解决方案：对 klk 站点上的所有页面添加 pageview 埋点，APP 在进行点击关联时加上 pageview 事件，这样若点击的 module/item 没有埋点，APP 也能关联到 page 级别的 spm。若所有页面都已经有 pageview 埋点，则 click_spm 的合理性和可解释性将极大增强。
对于手势返回，系统暂时无法监测到，如 A-B-A 的场景，第二个 A 页面拿到的 click_spm 会是第一个 A 页面的点击事件。
解决方案：APP 看下技术方案，是否可能给手势返回加一个 custom 事件埋点。
Web 设计方案
在 web 中传递 click_spm 和 click_id，最直观和主流的做法是通过 url 参数传递。但是由于这样做会影响网站的 SEO 表现，因此我们只能够采取其他方式实现。

目前的实现方案是：
用户产生点击时，将点击事件的 spm 和 click_id 记录到浏览器的 local storage 中。
用户在其他页面上发生的事件，会去查找 local storage 中记录的 click_id 和 click_spm。
这样在一般情况下，click_id 和 click_spm 实现了不同页面乃至不同 tab 之间的传递。
这个方案可能存在的问题是：
用户的点击链路更加复杂时，如打开多个 tab，或者在当前 tab 中再次点击刷新页面时，click_spm 和 click_id 的存储可能产生错乱的情况。
解决方案：待梳理。

与 in-house tracking 的结合
前端实验
前端的每条埋点数据都会将用户命中的所有实验信息带上，因此不需要 Kepler AB 项目的额外支持。
后端实验
Kepler 已经增加了对后端实验的支持，为了与 in-house tracking 项目结合进行分析，需要支持将后端实验命中数据 log 到大数据平台。
因此需要 Kepler 增加一个 Logging SDK，各服务在实验处理起始位置，通过调用 Kepler SDK 获得用户的实验命中结果；在实验处理结束位置，通过调用 Kepler Logging SDK 即可将用户命中实验的信息、以及业务自定义的字段发送到大数据平台。
不管检查到用户是掉入 variant、control 还是没有命中，都需要打 log，这样确保我们能够得到完整的数据，进行比较全面的分析
这个 log 在统一日志上报系统中有一个单独的 type=5，便于与其他类型的日志分开便于后续分析
业务自定义参数可能在查询用户命中结果的请求返回之后才生成，根据实验的实际情况也可能没有业务自定义参数，Logging SDK 需要两种情况都兼容

算法参数
模型打分
